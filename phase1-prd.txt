# Product Requirements Document

## Med Spa Revenue Rebooking System

### Phase 1 — Foundation + PMS Sync

Timeline: **Weeks 1–4**

---

# 1. Objective

The objective of Phase 1 is to establish the **technical foundation of the platform** and enable clinics to:

1. Sign up and authenticate
2. Connect their PMS (Aesthetic Record)
3. Sync patient data
4. View their patients in a dashboard
5. Manage messaging opt-outs

This phase ensures that the system can **ingest real clinic data**, which is required before building automation workflows.

---

# 2. Success Criteria

Phase 1 is considered successful if a clinic can:

1. Create an account
2. Connect their Aesthetic Record account
3. Sync patient data automatically
4. View patient list and details in the dashboard
5. Store patient communication preferences

Expected outcome:

```text
Clinic → connects PMS → patient data appears in dashboard
```

---

# 6. Database Architecture

The platform must support **multiple clinics using one system**.

Each row of data must belong to a clinic.

Key concept:

```

Every table must include:

```
clinic_id UUID

# 7. Row-Level Security

To isolate clinic data, PostgreSQL Row-Level Security must be enabled.

Example rule:

Only allow users to access rows where:

```
clinic_id = current_session_clinic
```

This prevents clinics from accessing each other's data.

# 9. PMS Integration — Aesthetic Record

Phase 1 supports **Aesthetic Record only**.

Integration responsibilities:

1. Authenticate clinic connection
2. Pull patient data
3. Pull treatment history
4. Pull appointments

---

## 9.1 Sync Method

Initial sync:

```
full patient import
```

Scheduled sync:

```
incremental updates
```

Recommended frequency:

```
every 6 hours
```

---

## 9.2 Sync Flow

```
Clinic connects Aesthetic Record
        ↓
API credentials stored
        ↓
Background job triggered
        ↓
Patient data pulled
        ↓
Patients stored in database
        ↓
Dashboard populated
```

Redis will handle background sync jobs.

# 11. Opt-Out Management

The system must support **communication preferences**.

Patients must be able to opt out of:

* SMS
* Email

Opt-out can occur through:

1. SMS reply ("STOP")
2. Manual admin update
3. Import from PMS

Rules:

If patient opted out → system cannot send messages.

# My Product Advice (Important)

Your Phase 1 is **well scoped**, but one thing will save you huge headaches later:

**Add an events table now.**

Even if you don’t use it yet.

Example:

```
events
------
id
clinic_id
event_type
entity_id
payload
created_at
```

Later automations trigger from events like:

```
appointment_cancelled
treatment_completed
botox_due
membership_expiring
```

If you add this now, Phase 2 becomes **much easier**.
